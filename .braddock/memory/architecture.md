## 1. Architectural Vision

The Nexus AI is built as a **modular AI-native monolith** — a single deployable backend (NestJS) and a single frontend (Next.js) — with internal boundaries so sharp that future extraction into services requires only plumbing changes, not rewrites.

### Guiding Principles

1. **Model-Agnostic Core** — Every AI interaction flows through an abstraction (`LLMProvider`, `EmbeddingProvider`). Swapping Ollama for OpenAI/Anthropic is a configuration change, not a refactor.
2. **Local-First, Cloud-Capable** — The default stack (Node.js, PostgreSQL+pgvector, Redis, Ollama) costs $0 to develop. Cloud providers are pluggable for demos.
3. **Observability as a First-Class Layer** — Structured logging, tracing, and evaluation are not add-ons; they are embedded in every flow and exposed via APIs and UI.
4. **Prompts as Versioned Assets** — Prompt templates live in code (TypeScript), versioned by Git, and loaded at runtime. No prompt drift in production.
5. **Simplicity over Distribution** — One backend process, one database, one cache. No Kubernetes, no microservices, no event buses. Complexity is reserved for AI behavior, not infrastructure.

#### 2.2.1 Design System Layer

The frontend implements a **dark-first, token-driven design system** defined in `.braddock/memory/design-guideline.md`.

| **Primitives** | `src/components/primitives/` | Atomic components: Pill, Button, Card, IconButton. Consume design tokens exclusively. |
| **Data Display** | `src/components/data-display/` | MetricCard, FeaturedMetricCard, AIInsightCard, DonutChart, StripedBar, CitationCard, StreakCard. |
| **AI Components** | `src/components/ai/` | Orb (pulse + glow + star particles), StarField, ChatInput, QuickActions, AgentTimeline. |
| **Layout** | `src/components/layout/` | TopNav, SectionIntro, FigmaBoundingBox (ornament), PipelineSteps (zigzag timeline). |
| **Font** | Google Fonts / self-hosted | Inter (sans), JetBrains Mono (mono), Instrument Serif (italic, rare). |
| **Icons** | `lucide-react` | Line-style icons, stroke 1.75px, always inside circular containers (32px or 40px). |
| **Charts** | `recharts` | Donut, bar, line charts styled with design tokens. Visx for custom visualizations if needed. |

**Key architectural constraints:**
- No light mode. The system is dark-only (`bg.canvas` = `#000000`).
- No arbitrary values in Tailwind classes — all visual properties must reference design tokens.
- All icons must be wrapped in circular containers with contrasting backgrounds for visibility on pure-black canvas.
- Orb AI and Featured AI Card are signature components; their presence is required on primary screens per the visual checklist.

**PostgreSQL Schema (MVP):**

```sql
-- Documents table
CREATE TABLE documents (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    filename TEXT NOT NULL,
    mime_type TEXT NOT NULL,
    status TEXT NOT NULL CHECK (status IN ('pending', 'processing', 'completed', 'failed')),
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Chunks table (references documents)
CREATE TABLE chunks (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    content TEXT NOT NULL,
    chunk_index INTEGER NOT NULL,
    metadata JSONB DEFAULT '{}',
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Vector table (pgvector extension required)
CREATE TABLE embeddings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    chunk_id UUID NOT NULL REFERENCES chunks(id) ON DELETE CASCADE,
    embedding VECTOR(1536), -- dimension varies by model; default to 1536 (OpenAI)
    model TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_embeddings_vector ON embeddings USING ivfflat (embedding vector_cosine_ops);

-- Observability logs
CREATE TABLE request_logs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    correlation_id TEXT NOT NULL,
    query TEXT,
    prompt TEXT,
    response TEXT,
    provider TEXT NOT NULL,
    model TEXT NOT NULL,
    latency_ms INTEGER,
    tokens_input INTEGER,
    tokens_output INTEGER,
    error TEXT,
    retrieved_chunk_ids UUID[],
    created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_request_logs_correlation ON request_logs(correlation_id);
CREATE INDEX idx_request_logs_created ON request_logs(created_at);

-- Evaluation runs
CREATE TABLE evaluation_runs (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    provider TEXT NOT NULL,
    model TEXT NOT NULL,
    status TEXT NOT NULL CHECK (status IN ('pending', 'running', 'completed', 'failed')),
    metrics JSONB DEFAULT '{}',
    started_at TIMESTAMPTZ DEFAULT NOW(),
    completed_at TIMESTAMPTZ
);

-- Evaluation results (per question)
CREATE TABLE evaluation_results (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    run_id UUID NOT NULL REFERENCES evaluation_runs(id) ON DELETE CASCADE,
    question TEXT NOT NULL,
    expected_answer TEXT,
    generated_answer TEXT,
    relevance_score NUMERIC,
    consistency_score NUMERIC,
    grounding_score NUMERIC,
    latency_ms INTEGER,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

## 3. Flows

### 3.1 Document Ingestion Flow

```text
User uploads file (UI)
    │
    ▼
Next.js API Route receives multipart/form-data
    │
    ▼
POST /api/ingestion → IngestionController
    │
    ▼
IngestionService:
  1. Save file metadata to documents table (status = 'pending')
  2. Enqueue BullMQ job on `ingestion` queue
    │
    ▼
IngestionWorker (BullMQ):
  1. Read file from temp storage
  2. Parse text (PDF → text, TXT → raw, MD → raw)
  3. Clean and normalize whitespace
  4. Chunk text (configurable size/overlap via Settings)
  5. Save chunks to chunks table
  6. Enqueue one EmbeddingWorker job per batch of chunks
  7. Update document status = 'processing'
    │
    ▼
EmbeddingWorker (BullMQ):
  1. Fetch chunk texts
  2. Call EmbeddingService → active provider (Ollama / OpenAI)
  3. Store vectors in embeddings table
  4. On completion of all chunks: update document status = 'completed'
  5. On failure: retry with exponential backoff (max 3); final fail → status = 'failed'
```

**Key Details:**

- Concurrency limit: 3 embedding jobs running at once (configurable).
- Retry policy: 3 attempts, exponential backoff starting at 2s.
- File size cap: 10MB per file (MVP guardrail).

### 3.2 Query / RAG Flow

```text
User submits chat message
    │
    ▼
POST /api/chat → ChatController
    │
    ▼
TraceMiddleware injects correlation_id (UUID v4)
    │
    ▼
ChatService:
  1. Check Redis cache (hash of query + session context)
     ├─ Cache HIT → return cached response, log hit
     └─ Cache MISS → continue
    │
    ▼
  2. Load session memory from Redis (last N turns)
    │
    ▼
  3. Call GuardrailsService → validate input (length, profanity, injection patterns)
     ├─ FAIL → return error, log rejection
     └─ PASS → continue
    │
    ▼
  4. Call AgentService (Decision Graph):
     ├─ "Direct Answer" → skip retrieval, call LLM directly
     └─ "RAG Required"  → continue to retrieval
    │
    ▼
  5. RetrievalService:
     a. Generate embedding for query (EmbeddingService)
     b. pgvector similarity search: SELECT chunk_id, content FROM embeddings ORDER BY embedding <=> $query_embedding LIMIT $top_k
     c. Optional: MMR diversification post-processing
    │
    ▼
  6. Build prompt via PromptManager:
     - System prompt (behavior, constraints)
     - Context prompt (retrieved chunks with source IDs)
     - User prompt (current query)
     - Memory prompt (recent conversation turns)
    │
    ▼
  7. Call LLMService → active provider
    │
    ▼
  8. GuardrailsService → validate output (JSON schema if structured mode; empty check)
    │
    ▼
  9. Cache response in Redis (TTL 1h)
    │
    ▼
 10. Log structured event to request_logs table:
     correlation_id, query, prompt, response, provider, model,
     latency_ms, tokens_input, tokens_output, retrieved_chunk_ids
    │
    ▼
Return response + source citations to UI
```

**Key Details:**

- Top-k default: 5 chunks.
- Session memory depth: last 10 turns (configurable).
- Cache key: SHA-256 hash of `provider:model:query:context_hash`.
- Latency target: < 3s cloud, < 8s local (p50).

### 3.3 Agent Decision & Multi-Step Workflow Flow

```text
ChatService delegates to AgentService
    │
    ▼
Decision Graph (LangGraph):
  ┌─────────────────┐
  │  START          │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ classify_query  │ ← LLM call with system prompt: "Classify as DIRECT or RAG"
  └────────┬────────┘
           │
     ┌─────┴─────┐
     ▼           ▼
┌─────────┐  ┌──────────┐
│ DIRECT  │  │ RAG      │
│ answer  │  │ required │
└────┬────┘  └────┬─────┘
     │            │
     ▼            ▼
┌─────────┐  ┌─────────────────────────────────────────────┐
│ END     │  │ Workflow Graph (multi-step)                 │
│         │  │  Step 1: retrieve (Tool: retrieval-tool)    │
│         │  │  Step 2: synthesize (LLM call with context) │
│         │  │  Step 3: validate (LLM self-check or rule)  │
│         │  │  Step 4: format (add citations, structure)  │
│         │  └─────────────────────────────────────────────┘
└─────────┘            │
                       ▼
                  ┌─────────┐
                  │ END     │
                  └─────────┘
```

**Key Details:**

- Decision graph is a single-node LangGraph state machine for simplicity.
- Workflow graph has 4 nodes: `retrieve`, `synthesize`, `validate`, `format`.
- State object (TypeScript interface) carries: `query`, `sessionId`, `retrievedChunks`, `draftResponse`, `finalResponse`, `correlationId`.
- Tool registry is injectable: agents receive `ToolRegistryService` and invoke tools by name.
- Conditional edges determine if validation passes or loops back to retrieve with a refined query.

### 3.4 Evaluation Flow

```text
User clicks "Run Evaluation" (UI) or calls CLI
    │
    ▼
POST /api/evaluation → EvaluationController
    │
    ▼
EvaluationService:
  1. Load eval dataset (`data/eval-dataset.json` — 20-50 Q&A pairs)
  2. Create evaluation_runs row (status = 'running')
  3. Enqueue BullMQ job on `evaluation` queue
    │
    ▼
EvalWorker (BullMQ):
  FOR each question in dataset:
    a. Call ChatService (reuses full RAG flow) with the question
    b. Capture: response, latency, provider, model, retrieved chunks
    c. Run heuristic scoring:
       - Relevance: overlap between expected and generated answer (token overlap / embedding similarity)
       - Consistency: check for contradictions via lightweight LLM call or rule-based check
       - Grounding: verify if generated answer contains facts present in retrieved chunks (heuristic)
    d. Insert row into evaluation_results
  END FOR
    │
    ▼
  Update evaluation_runs (status = 'completed', metrics = aggregated scores)
    │
    ▼
UI polls `/api/evaluation/:runId` and renders:
  - Per-question breakdown
  - Aggregate metrics (avg relevance, consistency, grounding, latency)
  - Side-by-side comparison if multiple runs exist
```

**Key Details:**

- Eval dataset is a static JSON file committed to the repo.
- Scoring is heuristic + optional lightweight LLM judge (configurable).
- Evaluation runs are isolated: they do not affect production caches or logs (use a distinct correlation_id prefix: `eval-{runId}`).
- Concurrency: 1 eval question at a time to avoid rate limits and ensure consistent latency measurement.

### 3.5 Observability / Logging Flow

```text
Every incoming HTTP request
    │
    ▼
TraceMiddleware (NestJS global middleware):
  1. Generate or propagate correlation_id (header x-correlation-id)
  2. Attach to AsyncLocalStorage context
    │
    ▼
LoggerService (Pino):
  - Automatically includes correlation_id in every log line
  - Outputs structured JSON: { timestamp, level, correlation_id, message, ...metadata }
    │
    ▼
At key lifecycle points, services call LoggerService explicitly:
  - ChatService: log "rag.start", "rag.retrieval.complete", "llm.request", "llm.response", "rag.complete"
  - IngestionWorker: log "ingestion.start", "ingestion.chunk.complete", "embedding.batch.complete"
  - EvalWorker: log "eval.question.start", "eval.question.complete"
    │
    ▼
Async background process (or on-demand job):
  - Persist high-signal logs to request_logs table (LLM calls, errors, retrievals)
  - Stdout JSON logs captured by Render/Railway log drains for debugging
    │
    ▼
Dashboard UI queries:
  - GET /api/dashboard/metrics (aggregated from request_logs)
  - GET /api/dashboard/logs (recent request_logs rows)
  - GET /api/dashboard/evaluations (evaluation_runs + results)
```

**Key Details:**

- Correlation ID links: user query → retrieval chunks → prompt → LLM response → error (if any).
- Log levels: `info` for normal flow, `warn` for guardrail triggers, `error` for exceptions.
- Sensitive data: no raw file contents logged; only metadata, chunk IDs, and truncated prompts.

## 4. Trade-offs

### 4.1 Monolith vs. Microservices

**Decision:** Modular monolith (single NestJS app).  
**Rationale:** The MVP has no independent scaling requirements per module. A monolith eliminates network overhead, simplifies local development, and keeps deployment to Render/Railway trivial. Modules are bounded so that extraction later (e.g., moving AI Core to a separate service) is a matter of adding HTTP/gRPC clients, not rewriting logic.  
**Cost:** Future scaling requires a rewrite if one module becomes a bottleneck; mitigated by keeping modules stateless.

### 4.2 pgvector vs. Dedicated Vector Database (Pinecone, Weaviate, Qdrant)

**Decision:** PostgreSQL + pgvector.  
**Rationale:** Zero additional infrastructure. The team already needs PostgreSQL for relational data; adding pgvector avoids a second database to manage, backup, and secure. For the MVP scale (< 10k chunks), `ivfflat` indexing is sufficient.  
**Cost:** pgvector lacks advanced hybrid search and may degrade at > 100k chunks without HNSW tuning. The trade-off is accepted for MVP; migration to a dedicated vector DB is documented as a post-MVP scaling strategy.

### 4.3 LangGraph vs. Custom Orchestration vs. Heavy LangChain

**Decision:** LangGraph for agent graphs, minimal LangChain elsewhere.  
**Rationale:** LangGraph provides explicit state management and visualizable graphs for multi-step workflows without forcing LangChain abstractions into every layer. Core services (LLM, RAG, retrieval) use thin native adapters (Ollama SDK, OpenAI SDK) to avoid framework lock-in.  
**Cost:** LangGraph is newer and has a smaller community than LangChain; debugging state transitions requires learning curve. Mitigated by keeping graphs simple (4 nodes max in MVP).

### 4.4 Local-First (Ollama) vs. Cloud-First

**Decision:** Ollama default for development; OpenAI/Anthropic selectable for demos.  
**Rationale:** Development cost must be $0. Local models enable rapid iteration without API keys or quota anxiety. A clean `LLMProvider` interface makes cloud switching a runtime config change.  
**Cost:** Local models are slower and less capable than cloud APIs. The hybrid strategy mitigates this: validate locally, polish demos with cloud.

### 4.5 NestJS vs. Express / Fastify Raw

**Decision:** NestJS.  
**Rationale:** The squad requires strong module boundaries, dependency injection, and built-in support for middleware, interceptors, and pipes — all of which enforce consistency across a multi-developer team. Express is too loose for a structured AI-native monolith.  
**Cost:** NestJS has a steeper learning curve for developers unfamiliar with decorators and DI containers. Mitigated by clear module conventions documented in the project.

### 4.6 Redis as Single Broker vs. Separate Message Queue (RabbitMQ, SQS)

**Decision:** Redis for cache, sessions, and BullMQ queues.  
**Rationale:** Redis is already required for sessions and caching. BullMQ provides robust job semantics (retries, delays, concurrency) on top of Redis without introducing a second infrastructure component.  
**Cost:** Redis becomes a single point of failure for both caching and background jobs. Mitigated by using Redis persistence (AOF) and keeping queue jobs idempotent so they can be re-enqueued after recovery.

### 4.7 TypeORM vs. Prisma

**Decision:** Prisma (recommended) or TypeORM.  
**Rationale:** Prisma offers superior type safety, migration tooling, and query performance for PostgreSQL. If the team prefers NestJS conventions, TypeORM is acceptable. The architecture document assumes Prisma for schema management and type-safe queries.  
**Cost:** Prisma has known issues with very complex raw queries (e.g., pgvector `<=>` operators). Mitigated by using Prisma for relational CRUD and raw SQL for vector search.

## 5. Risks

### 5.1 Local LLM Performance and Quality

**Risk:** Ollama models (e.g., Llama 3, Mistral 7B) may be too slow (> 8s latency) or produce low-quality responses on the evaluation dataset, damaging demo credibility.  
**Mitigation:** Hybrid strategy — always validate with cloud APIs before concluding that the system is flawed. Keep local models for cost-free development. Document model recommendations (e.g., `llama3:8b` as default) and provide a benchmark script.

### 5.2 pgvector Scaling Ceiling

**Risk:** As document volume grows, `ivfflat` indexes on pgvector may yield slow queries or high memory usage, especially with frequent inserts.  
**Mitigation:** Cap MVP ingestion to < 10k chunks. Add HNSW indexes if latency degrades. Document migration path to Pinecone/Qdrant for post-MVP growth.

### 5.3 LangGraph Complexity

**Risk:** LangGraph's state-machine paradigm may introduce bugs in state transitions, infinite loops in workflows, or opaque errors that are hard to debug.  
**Mitigation:** Keep graphs minimal (decision graph = 1 node; workflow graph = 4 nodes). Write unit tests for every node and edge condition. Log state snapshots at each transition using the correlation ID.

### 5.4 Redis as Single Point of Failure

**Risk:** If Redis fails, the system loses session memory, response cache, and all async job processing simultaneously.  
**Mitigation:** Enable Redis AOF persistence. Design jobs to be idempotent (re-processing a chunk is safe). Gracefully degrade: if Redis is unavailable, chat falls back to stateless mode and ingestion returns synchronous errors.

### 5.5 Async Queue Reliability

**Risk:** BullMQ jobs (ingestion, embedding, evaluation) may fail silently, stall, or overwhelm the Ollama server with concurrent embedding requests.  
**Mitigation:**

- Configure strict concurrency limits per worker (e.g., 3 embedding jobs max).
- Use exponential backoff with dead-letter queues after 3 retries.
- Expose queue status via `/api/health/queues` for monitoring.
- Add worker heartbeat checks and UI-level job status polling.

### 5.6 Prompt Drift and Version Management

**Risk:** As prompts evolve, subtle changes may degrade retrieval quality or agent behavior without detection.  
**Mitigation:** Prompts are TypeScript files in `src/prompts/`, versioned in Git. Every prompt change requires a PR. The evaluation dataset is re-run after prompt PRs to catch regressions.

### 5.7 Guardrail Bypasses

**Risk:** Lightweight guardrails (regex, length checks, basic JSON schema) may not catch sophisticated prompt injection or unsafe outputs.  
**Mitigation:** Accept MVP-level guardrails. Log all guardrail triggers for review. Post-MVP, integrate a dedicated guardrail service or heavier LLM-based moderation.

### 5.8 Evaluation Subjectivity

**Risk:** Heuristic scoring (token overlap, embedding similarity) may not reflect true answer quality, leading to false confidence in model performance.  
**Mitigation:** Combine automated metrics with a fixed human review sample (10 questions per run). Keep the dataset small and domain-specific so scoring is interpretable.
