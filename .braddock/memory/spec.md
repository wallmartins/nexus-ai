## 1. User Journeys

### 1.1 Journey: First-Time User Uploads a Document and Asks a Question
**User Goal:** Ground the system in proprietary knowledge and validate that it can answer based on uploaded content.

| Step | Touchpoint | Action | Emotion / Expectation | Potential Pain Point |
|------|------------|--------|----------------------|----------------------|
| 1 | Landing / Chat screen | User sees the **Chat Hero** with the **Orb AI** (purple gradient sphere + star particles + 2.4s pulse) and a 4-card **Quick Actions** grid below. A prominent lime primary Button reads "Upload Documents" | Curious, wants quick proof of value | Unclear where to start if CTA is buried |
| 2 | Document Upload screen | User drags a PDF into the **Upload Dropzone** (`surface.2` bg, `border.subtle`, `md` radius). On dragover, border shifts to `brand.purple.500` with a `glowPurple` shadow | Expects immediate feedback | Large files hang without progress indication |
| 3 | Document Upload screen | System renders a **Pipeline Timeline** with zigzag cards and dashed bezier connectors: **Ingestion** (`cat.orange`) → **Chunking** (`cat.purple`) → **Embedding** (`cat.lime`). In-process steps show a diagonal hatched pattern skeleton | Reassured by transparency | Vague "Processing..." without step breakdown |
| 4 | Toast notification | A lime-accented toast appears: "Document ready — 42 chunks indexed" with a **delta chip** showing `+100%` coverage vs. prior state | Confident to proceed | Notification dismissed too quickly |
| 5 | Chat screen | User types a domain-specific question into the **Chat Input** (`surface.2` bg, `xl` 28px radius, attach + mic icons left, lime send Button right) | Expects grounded, cited answer | Empty response or generic hallucination erodes trust |
| 6 | Chat screen | Answer streams in with inline **citation markers** `[1]`, `[2]` styled as subtle `brand.purple.400` links inside the assistant bubble | Engaged, wants to verify sources | Citations unclickable or missing |
| 7 | Chat screen | User expands the **Sources Panel** to preview retrieved chunks rendered as **Citation Cards** (thumbnail + doc name + meta + similarity bar) | Trust increases if chunks match question | Source panel shows irrelevant excerpts |
| 8 | Chat screen | User asks a follow-up referencing "the previous answer" | Expects session memory to work | Context lost; user must repeat earlier details |

**Key Design Decision:** The empty state of the Chat screen must always surface the upload action so first-time users are not blocked. The Orb AI and Quick Actions grid provide visual anchors that reduce cognitive load.

### 1.2 Journey: User Compares Local vs. Cloud LLM Responses
**User Goal:** Validate cost/quality trade-offs between an Ollama model and a cloud API.

| Step | Touchpoint | Action | Emotion / Expectation | Potential Pain Point |
|------|------------|--------|----------------------|----------------------|
| 1 | Settings screen | User selects "Comparison Mode" via a **Pill** toggle and picks two providers from **Provider Selector** cards. Each selected provider shows a categorical color dot (`cat.blue` for cloud, `cat.lime` for local) | Excited to see differences | UI does not clearly indicate which model is "A" vs "B" |
| 2 | Chat screen | User enters a question and toggles "Run Both Models" (accent purple Pill) before pressing the lime send Button | Expects side-by-side layout | Accidentally sends to only one model |
| 3 | Chat screen | Two answer panes appear in a **50/50 split-pane** layout on desktop (`border.subtle` divider, draggable handle). Below `1024px`, stacked tabs with a lime active indicator | Wants to scan differences quickly | Layout breaks on smaller viewports |
| 4 | Chat screen / Citations | Each pane shows its own **Citation Cards** and a **Metric Card** latency badge with a delta chip (`-12%` vs. baseline) | Wants objective metrics | Latency hidden in a tooltip |
| 5 | Chat screen | User clicks the **icon Button** (40px circle, ghost variant) to "Copy" or "thumbs-up" on the preferred pane | Wants to record preference | No way to save preference for future reference |
| 6 | Observability Dashboard | User later reviews the comparison run ID in the **Request Log Table**. A **Donut Chart** (lime + purple segments) shows token split between providers | Needs cost audit trail | Comparison runs not tagged distinctly in logs |

**Key Design Decision:** Comparison mode uses a split-pane layout on desktop (50/50) and stacked tabs on viewports below `1024px`. Each pane is color-coded with a subtle left border (A = `text.secondary`, B = `brand.purple.500`) to prevent confusion. Delta chips on latency metrics make trade-offs scannable at a glance.

### 1.3 Journey: User Runs an Evaluation and Reviews Results
**User Goal:** Measure system quality on a fixed dataset and identify regressions.

| Step | Touchpoint | Action | Emotion / Expectation | Potential Pain Point |
|------|------------|--------|----------------------|----------------------|
| 1 | Evaluation screen | User clicks the lime primary Button "New Evaluation Run" | Expects configuration options | Accidentally triggers run with wrong model |
| 2 | Evaluation modal | Modal surface (`surface.3` bg, `lg` 20px radius, `float` shadow). User selects LLM provider, embedding model, and top-k value via dropdowns; clicks "Start" | Feels in control | Missing validation on required fields |
| 3 | Evaluation screen | A job card appears with live progress (e.g., 12/50 questions). The card uses a **diagonal hatched pattern** overlay while `running` to signal in-process state | Patient if progress is visible | Stuck at 0% with no error message |
| 4 | Evaluation screen | Run completes. A **Featured AI Card** (gradient purple bg + star particles + hatched bars) displays the overall AI score. Below, **Metric Cards** show Relevance, Consistency, Grounding with **delta chips** (`+4%`, `-1%`) vs. last run | Wants summary at a glance | Scores buried in a table |
| 5 | Evaluation screen | User drills into a specific question row. A **Timeline Widget** shows the pipeline stage where the failure occurred, highlighted with a lime event pin | Expects to see expected vs. actual answer, retrieved chunks, and per-metric verdict | Difficult to map failure back to retrieval vs. generation |
| 6 | Evaluation screen | User exports a JSON report via a ghost Button or copies a shareable run link | Wants reproducibility | Export missing correlation IDs |

**Key Design Decision:** Evaluation results use a **scorecard header** (hero metric + delta chip + context caption) above a sortable, filterable table. Failed rows are highlighted with a `status.warning` (`#F7B955`) border and warning icon, not just red text, to distinguish metric failures from system errors. The Featured AI Card establishes visual dominance for the overall health score.

### 1.4 Journey: User Investigates a Slow or Failed Query via Observability Dashboard
**User Goal:** Debug latency spikes or errors using structured traces.

| Step | Touchpoint | Action | Emotion / Expectation | Potential Pain Point |
|------|------------|--------|----------------------|----------------------|
| 1 | Observability Dashboard | User sees a **Chart Card** (title + filter dropdown + Recharts body + dot legend) showing p50 latency with an anomalous spike at 14:32. A **Streak Card** floats nearby showing eval pipeline uptime streak | Alerted but not overwhelmed | Chart lacks tooltips or exact values |
| 2 | Observability Dashboard | User filters the **Request Log Table** by time range and status "Error" using Pill filters. Active filter = lime bg | Expects fast filtering | Filter state lost on refresh |
| 3 | Observability Dashboard | User clicks a request row to open the **Trace Detail drawer** (`surface.2` bg, `float` shadow, slides in with `base` 220ms `emphasized` easing) | Wants end-to-end visibility | Drawer loads slowly because payload is huge |
| 4 | Trace Detail drawer | **Gantt-style timeline** shows: Receive → Retrieve (120ms, 5 chunks, `cat.lime`) → LLM Call (4.2s, 847 tokens, `cat.purple`) → Respond. Each bar length is proportional to duration | Immediately spots LLM call as bottleneck | Timeline lacks collapsible sections |
| 5 | Trace Detail drawer | User expands the "Prompt" and "Response" **accordion** (`surface.1` bg, `sm` 10px radius). Code blocks use **Geist Mono** for readability | Needs auditability | JSON escaping makes text unreadable |
| 6 | Trace Detail drawer | User copies the `correlation-id` (rendered in Geist Mono, `text.muted`) to share with a teammate via a one-click icon Button | Expects one-click copy | ID truncated or unselectable |

**Key Design Decision:** The trace timeline is rendered as a horizontal Gantt-style bar inside the drawer. Each stage is color-coded using categorical tokens (**Retrieval** = `cat.lime`, **LLM** = `cat.purple`, **Error** = `cat.pink`) and proportional to duration, enabling instant visual diagnosis.

## 2. Modules

### 2.1 Backend (NestJS)

| Module | Purpose | Key Services / Controllers | Dependencies |
|--------|---------|---------------------------|--------------|
| **AppModule** | Root module. Bootstrap, global pipes, interceptors, exception filters. | `AppController`, `AppService` | ConfigModule |
| **ConfigModule** | `@nestjs/config` global configuration. Validates `process.env` against a typed schema (e.g., `zod`) on startup. | `ConfigService` | None |
| **DatabaseModule** | TypeORM (or chosen ORM) connection to PostgreSQL with pgvector extension. Exposes repositories. | `DatabaseService` | ConfigModule |
| **IngestionModule** | Handles document upload, parsing, cleaning, chunking, and dispatching embedding jobs. | `IngestionController`, `IngestionService`, `DocumentParserService`, `ChunkingService` | DatabaseModule, QueueModule, ObservabilityModule |
| **RAGModule** | Vector search and retrieval logic. Owns chunk storage and top-k/MMR queries against pgvector. | `RetrievalService`, `ChunkRepository` | DatabaseModule, ConfigModule |
| **LLMModule** | Model-agnostic LLM provider abstraction, prompt composition, guardrails, and token tracking. | `LlmService`, `OllamaProvider`, `OpenAiProvider`, `AnthropicProvider`, `PromptRegistry`, `GuardrailService` | ConfigModule, ObservabilityModule |
| **AgentModule** | LangGraph workflows, decision agents, and tool registry. | `AgentService`, `DecisionAgent`, `SynthesisWorkflow`, `ToolRegistry`, `RetrievalTool`, `CalculatorTool` | LLMModule, RAGModule, MemoryModule, ObservabilityModule |
| **MemoryModule** | Redis-backed session memory and response cache. | `MemoryService`, `CacheService` | ConfigModule |
| **ObservabilityModule** | Structured logging, correlation ID propagation, metrics aggregation, and trace linking. | `LoggerService`, `CorrelationMiddleware`, `MetricsService` | ConfigModule, DatabaseModule |
| **EvaluationModule** | Dataset loading, on-demand evaluation runs, model comparison, and score computation. | `EvaluationController`, `EvaluationService`, `DatasetLoader`, `ScoringService` | DatabaseModule, QueueModule, LLMModule, RAGModule, ObservabilityModule |
| **QueueModule** | BullMQ queue registration, producers, and consumers. | `QueueService`, `IngestionConsumer`, `EmbeddingConsumer`, `EvaluationConsumer` | ConfigModule, DatabaseModule, LLMModule, RAGModule |
| **CommonModule** | Shared utilities: DTOs, custom pipes (`FileSizeValidationPipe`), decorators, and base exception classes. | `FileSizeValidationPipe`, `ZodValidationPipe` | None |

**Interface/API Patterns**
- All modules expose public Services that may be imported by other modules.
- Cross-module communication is via service methods (synchronous) or BullMQ jobs (asynchronous) only. No direct repository sharing outside owning module.
- LLM providers implement a common `ILLMProvider` interface: `generate(prompt: PromptContext, options: LlmOptions): Promise<LlmResponse>`.
- Agents implement a common `IAgent` interface: `invoke(state: AgentState): Promise<AgentState>`.

### 2.2 Frontend (Next.js — App Router)

| Route / Area | Purpose | Key Components | State Approach |
|--------------|---------|---------------|--------------|
| **(chat)** `/chat` | Primary conversational UI. | `ChatLayout`, `ChatInterface`, `MessageBubble`, `SourcePanel`, `InputBar` | Zustand for active session + React Query for message history |
| **(chat)** `/chat/[sessionId]` | Deep-link to a specific session. | `ChatInterface`, `MessageList` | React Query to hydrate session |
| **(upload)** `/documents` | Document management and upload. | `UploadDropzone`, `DocumentList`, `IngestionStatusBadge`, `DocumentCard` | React Query for server state; optimistic updates on upload |
| **(dashboard)** `/dashboard` | Observability and metrics view. | `DashboardLayout`, `RequestLogTable`, `LatencyChart`, `TokenUsageChart`, `ErrorRateChart` | React Query for polling metrics |
| **(eval)** `/evaluations` | Evaluation runs and comparisons. | `EvaluationList`, `RunTriggerButton`, `ComparisonTable`, `ScoreBadge` | React Query |
| **(settings)** `/settings` | System configuration UI. | `SettingsForm`, `ProviderSelector`, `ChunkingConfig`, `EmbeddingModelSelector` | Zustand for draft form state; React Query for mutate |
| **API Routes** `/api/*` | Next.js BFF (Backend-for-Frontend) proxies to NestJS API if needed for auth or file handling; otherwise direct fetch. | Route handlers | — |

**State Management**
- **Server State**: TanStack Query (React Query) with stale-time of 30s for logs, 5s for active ingestion jobs.
- **Client State**: Zustand for lightweight global UI state (active session ID, settings draft, sidebar open/close).
- **No Redux** — simplicity first.

## 3. Design System & Component Library

### 3.1 Token Architecture

Tokens are organized in three layers under `src/design-tokens/`:

| Layer | Purpose | Example |
|-------|---------|---------|
| **Primitives** | Raw values: hex codes, font stacks, base scales | `color-black: #000000`, `space-4: 4px` |
| **Semantic** | Meaningful mappings: background, text, status | `bg-canvas: var(--color-black)`, `text-primary: #FFFFFF` |
| **Component** | Component-specific aliases | `button-primary-bg: var(--brand-lime-400)` |

**Color Primitives**
- **Background**: `bg.canvas` `#000000`, `bg.elevated` `#0A0A0A`
- **Surface**: `surface.1` `#111111`, `surface.2` `#161616`, `surface.3` `#1E1E1E`
- **Border**: `border.subtle` `rgba(255,255,255,0.06)`, `border.default` `rgba(255,255,255,0.10)`, `border.strong` `rgba(255,255,255,0.16)`
- **Brand**: `brand.lime.400` `#C5F547` (primary action), `brand.purple.400` `#A78BFA` (secondary), `brand.purple.500` `#8B5CF6` (AI intelligence)
- **Text**: `text.primary` `#FFFFFF`, `text.secondary` `#C7C7C7`, `text.muted` `#8A8A8A`, `text.disabled` `#555555`
- **Status**: `status.success` `#C5F547`, `status.warning` `#F7B955`, `status.danger` `#F47266`, `status.info` `#A78BFA`
- **Categorical**: `cat.orange` `#F97316` (Ingestion), `cat.purple` `#8B5CF6` (Embedding), `cat.lime` `#C5F547` (Retrieval), `cat.yellow` `#FACC15` (Orchestration), `cat.blue` `#3B82F6` (Evaluation), `cat.pink` `#EC4899` (Guardrails)

**Typography Primitives**
- **Sans**: Inter (`--font-sans`) — primary UI font. Weights: Regular (400), Medium (500), Semibold (600). No Bold/Black.
- **Mono**: JetBrains Mono (`--font-mono`) — exclusively for technical data: IDs, timestamps, numeric metrics, paths, code, correlation IDs, similarity scores. Never for UI labels, body text, buttons, or navigation.
- **Serif**: Instrument Serif (italic, via Google Fonts) — rare expressive use only in cover/hero headings for one keyword. Never in functional UI.
- **Headline Pattern**: Mix Regular (400) + Semibold (600) in the same line (e.g., "Grounded answers from **your knowledge base**").

**Spacing Scale (Base 4px, Non-linear)**
`4, 8, 12, 16, 20, 24, 32, 40, 48, 64, 80, 96, 128px`

**Border Radius Scale**
`xs: 6px, sm: 10px, md: 14px, lg: 20px, xl: 28px, 2xl: 36px, pill: 9999px`

**Shadows (Flat Design; Floating Only)**
- `float`: `0 24px 48px -12px rgba(0,0,0,0.6), 0 0 0 1px rgba(255,255,255,0.05)`
- `glowLime`: `0 0 40px rgba(197,245,71,0.35)`
- `glowPurple`: `0 0 60px rgba(139,92,246,0.45)`

**Motion Tokens**
- Durations: `instant: 80ms`, `fast: 150ms`, `base: 220ms`, `slow: 380ms`, `deliberate: 640ms`
- Easing: `standard: cubic-bezier(0.2,0,0,1)`, `emphasized: cubic-bezier(0.05,0.7,0.1,1)`, `decelerate: cubic-bezier(0,0,0,1)`

### 3.4 Asset Guidelines

**Logo Monogram**
- 1–2 letters, geometric sans, very bold weight (Semibold 600 max, visually heavy), inside a rounded square container (`md` radius, `surface.2` bg).

**Star Particles**
- 12–18 white dots, opacity oscillation animation (2–3s cycle)
- Density: ~1 dot per 40×40px area
- Usage: Orb AI background, Featured AI Card overlay, loading states

**Hatched Diagonal Pattern**
- CSS: `repeating-linear-gradient(135deg, currentColor 0px, currentColor 1.5px, transparent 1.5px, transparent 6px)`
- Usage: In-process skeletons, placeholder charts, job cards with `running` status

**Figma Ornaments**
- Purple bounding box with resize handles around section headings
- Max 1 per viewport section; never stack multiple ornaments
- Usage: Hero headings, major section intros

**Icon System**
- **Library**: `lucide-react` — line icons, stroke 1.75px, sizes 12/14/16/18/22.
- **Rule**: Every icon lives inside a circular container (32px or 40px) with a `surface.2` or categorical-tinted background. Never use raw icons without a container on dark surfaces.
- **Size**: 20px default inside 40px container; 16px inside 32px container.
- **Color**: `currentColor` by default, inheriting from parent text.

### 3.6 Voice & Copywriting

**Principles**
- Precise, technical, honest. Never exaggerated.
- Data > adjectives. "94% grounding" > "excellent grounding".
- Correct terms: RAG, embedding, chunk, retriever, guardrail, grounding, hallucination, correlation ID, latency p95.
- Lowercase for technical labels: `p50 latency`, `tokens in/out`.
- Title case for page headers: "Enterprise Pricing Model".

**Microcopy Examples**
| Situation | Good ✓ | Bad ✗ |
|-----------|--------|-------|
| Primary button | "Run evaluation", "Upload document" | "Submit", "Go" |
| Empty state | "No documents yet. Drop files to ingest." | "Nothing here!" |
| Error | "Ingestion failed · timeout at chunk 14" | "Oops! Something went wrong." |
| Success toast | "Deployed · v23 live" | "Success!" |
| Loading | "Retrieving 5 chunks…" | "Loading…" |

**Prohibitions**
- Never use "AI magic", "powered by AI", "intelligent".
- Never use emoji in functional UI.
- Never use overly intimate 2nd person ("Hey!", "Let's do this!"). Tone is professional tool.

## 4. Functional Requirements

### 4.1 RAG Pipeline
- **FR-RAG-001**: The system shall accept unstructured document uploads (PDF, TXT, Markdown) via REST API and web UI, with a per-file maximum size of 10MB.
- **FR-RAG-002**: The system shall parse uploaded documents to extract raw text, strip non-content noise (headers, footers, repeated elements), and normalize whitespace.
- **FR-RAG-003**: The system shall split parsed text into chunks using a configurable strategy (default: recursive character splitting) with configurable chunk size and overlap.
- **FR-RAG-004**: The system shall generate vector embeddings for each chunk asynchronously via a BullMQ worker, using a model-agnostic embedding provider.
- **FR-RAG-005**: The system shall store chunks and their corresponding embeddings in PostgreSQL with the pgvector extension.
- **FR-RAG-006**: The system shall execute top-k similarity search against the vector store given a query embedding, returning the most relevant chunks.
- **FR-RAG-007**: The system shall support optional Maximal Marginal Relevance (MMR) diversification to reduce redundancy in retrieved chunks.
- **FR-RAG-008**: The system shall track and expose the ingestion lifecycle status of each document: `pending`, `parsing`, `chunking`, `embedding`, `completed`, `failed`.
- **FR-RAG-009**: The system shall allow deletion of a document and its associated chunks and embeddings via API/UI.

### 4.2 LLM Orchestration
- **FR-LLM-001**: The system shall expose a unified `LLMProvider` interface that abstracts all model-specific implementations.
- **FR-LLM-002**: The system shall support at least three providers: Ollama (local), OpenAI, and Anthropic.
- **FR-LLM-003**: The system shall store prompt templates as versioned TypeScript assets in `src/prompts/` (or equivalent), committed to source control.
- **FR-LLM-004**: The system shall compose final prompts from system instructions, user input, retrieved context, and session memory.
- **FR-LLM-005**: The system shall request structured outputs (JSON mode or JSON Schema) from providers that support it, and validate the output against the expected schema.
- **FR-LLM-006**: The system shall apply input guardrails to reject empty, excessively long, or malformed user queries before LLM invocation.
- **FR-LLM-007**: The system shall apply output guardrails by validating that LLM responses conform to declared TypeScript types or JSON schemas; if validation fails, retry once or return a structured error.
- **FR-LLM-008**: The system shall record token usage (input and output) for every LLM call.
- **FR-LLM-009**: The system shall record end-to-end latency for every LLM call, from request dispatch to parsed response.

### 4.3 Agent Layer
- **FR-AGT-001**: The system shall implement a Decision Agent that classifies each user query to determine whether RAG retrieval is required or if the query can be answered directly.
- **FR-AGT-002**: The system shall implement multi-step workflows using LangGraph (preferred) or LangChain, represented as a directed state graph.
- **FR-AGT-003**: The system shall provide a default Synthesis Workflow with steps: `retrieve` → `synthesize` → `validate`.
- **FR-AGT-004**: The system shall maintain an extensible Tool Registry, starting with a Retrieval Tool (vector search) and a Calculator Tool (math evaluation).
- **FR-AGT-005**: The system shall allow agents to invoke tools dynamically based on workflow state and return tool outputs back into the graph state.
- **FR-AGT-006**: The system shall persist intermediate agent state during a workflow execution so that steps are resumable and traceable.

### 4.4 Memory & State Management
- **FR-MEM-001**: The system shall maintain short-term session memory per conversation, stored in Redis with a unique session ID.
- **FR-MEM-002**: The system shall retain the last N messages per session (default N=10, configurable), including user and assistant roles.
- **FR-MEM-003**: The system shall inject the retained session history into the prompt context for multi-turn conversations.
- **FR-MEM-004**: The system shall provide a Redis-backed response cache to avoid redundant LLM calls for identical or near-identical queries.
- **FR-MEM-005**: The system shall generate cache keys from a deterministic hash of the query text, selected LLM provider, model name, and retrieval fingerprint (top-k chunk IDs).
- **FR-MEM-006**: The system shall enforce a configurable TTL on cached responses (default 1 hour).

### 4.5 Observability
- **FR-OBS-001**: The system shall emit structured JSON logs for every incoming request and outgoing response.
- **FR-OBS-002**: The system shall log the full prompt sent to the LLM, the raw response received, latency in milliseconds, token usage, and any errors.
- **FR-OBS-003**: The system shall generate a unique correlation ID for each user request and propagate it through retrieval, LLM calls, agent steps, and queue jobs.
- **FR-OBS-004**: The system shall persist log entries with correlation IDs so that an entire request lifecycle can be reconstructed.
- **FR-OBS-005**: The system shall expose a dashboard UI displaying recent requests with their correlation IDs, statuses, and latencies.
- **FR-OBS-006**: The system shall expose aggregate metrics on the dashboard: average latency (p50/p95), total token consumption, and error rate over selectable time windows.
- **FR-OBS-007**: The system shall allow filtering and time-range querying of logs via API for debugging and audit purposes.

### 4.6 Evaluation Pipeline
- **FR-EVL-001**: The system shall load a static evaluation dataset of 20–50 domain-relevant questions from a committed file in the repository.
- **FR-EVL-002**: The system shall allow on-demand execution of an evaluation run via a REST endpoint and a UI button.
- **FR-EVL-003**: The system shall support evaluating multiple LLM providers/models within a single run.
- **FR-EVL-004**: The system shall compute a retrieval relevance score (precision@k) for each question based on the overlap between retrieved chunks and expected source references.
- **FR-EVL-005**: The system shall compute a consistency score by checking semantic similarity or exact-match heuristics between generated answers and reference answers.
- **FR-EVL-006**: The system shall compute a grounding score by verifying that generated answers contain citations linked to retrieved chunks.
- **FR-EVL-007**: The system shall record per-question latency and token consumption during evaluation.
- **FR-EVL-008**: The system shall present side-by-side model comparison in the UI, showing scores, latency, and cost per model.

### 4.7 Web Interface
- **FR-UI-001**: The system shall provide a chat UI where users can send messages and view assistant responses.
- **FR-UI-002**: The system shall display source citations (chunk references) alongside assistant answers, linked to originating documents.
- **FR-UI-003**: The system shall provide a drag-and-drop document upload interface with real-time or near-real-time ingestion status feedback.
- **FR-UI-004**: The system shall list uploaded documents with their current ingestion status, size, and upload timestamp.
- **FR-UI-005**: The system shall provide an observability dashboard with tables and charts for request logs, latency distributions, token usage, and error rates.
- **FR-UI-006**: The system shall provide a settings panel to select the active LLM provider and model.
- **FR-UI-007**: The system shall provide a settings panel to configure chunk size, chunk overlap, and embedding model.
- **FR-UI-008**: The system shall provide a settings panel to configure session memory depth (N) and response cache TTL.
- **FR-UI-009**: The system shall provide a responsive web layout optimized for desktop and functional on tablet viewports.

### 4.8 Async Job Processing
- **FR-ASY-001**: The system shall enqueue document ingestion tasks (parsing, chunking) on a dedicated BullMQ queue.
- **FR-ASY-002**: The system shall enqueue embedding generation tasks on a dedicated BullMQ queue, consumed asynchronously by workers.
- **FR-ASY-003**: The system shall enqueue evaluation pipeline tasks on a dedicated BullMQ queue.
- **FR-ASY-004**: The system shall implement retry logic with exponential backoff for failed jobs, with a maximum of 3 attempts.
- **FR-ASY-005**: The system shall enforce concurrency limits per queue (e.g., embedding workers max 2 concurrent, ingestion max 1 concurrent) to prevent resource exhaustion.
- **FR-ASY-006**: The system shall expose job status (pending, active, completed, failed) via API and reflect it in the UI.
- **FR-ASY-007**: The system shall route jobs that exhaust all retries to a dead-letter state (recorded in Redis/BullMQ and database) for manual inspection.

## 5. Non-Functional Requirements

### 5.1 Performance
- **NFR-PER-001**: The p50 latency for a cloud LLM chat response (including retrieval) shall be less than 3 seconds under normal load.
- **NFR-PER-002**: The p50 latency for a local Ollama LLM chat response shall be less than 8 seconds under normal load.
- **NFR-PER-003**: The p95 latency for top-k vector retrieval shall be less than 200ms for datasets up to 10,000 chunks.
- **NFR-PER-004**: The system shall process a standard document upload (≤ 10MB) end-to-end (upload → parse → chunk → embed → store) within 2 minutes.
- **NFR-PER-005**: The system shall support at least 10 concurrent chat sessions without response degradation.

### 5.2 Scalability
- **NFR-SCA-001**: The pgvector datastore shall support up to 10,000 chunks in the MVP without requiring reindexing, using IVFFlat or HNSW indexing strategies.
- **NFR-SCA-002**: The backend API shall be horizontally scalable as a stateless container on Render/Railway; no local disk state shall be required for API instances.
- **NFR-SCA-003**: BullMQ workers shall be deployable as separate processes or services from the API server, allowing independent scaling of queue consumers.

### 5.3 Reliability
- **NFR-REL-001**: All external LLM API calls shall implement retries with exponential backoff (up to 2 attempts) on transient errors (5xx, timeout).
- **NFR-REL-002**: If an LLM provider call fails definitively, the system shall return an explicit, user-friendly error response and log the failure; silent failures are prohibited.
- **NFR-REL-003**: If retrieval returns zero relevant chunks for a RAG-dependent query, the system shall inform the user that no relevant context was found rather than generating an ungrounded answer.
- **NFR-REL-004**: If Redis becomes unavailable, the system shall degrade gracefully: chat continues without session memory and response caching, and queue jobs remain queued in Redis until recovery.
- **NFR-REL-005**: All queued jobs shall persist their state in Redis and BullMQ until completed or moved to a dead-letter record; in-flight jobs shall not be lost on worker restart.

### 5.4 Security
- **NFR-SEC-001**: No secrets, API keys, or credentials shall be hardcoded in source code; all secrets shall be injected via environment variables.
- **NFR-SEC-002**: LLM provider API keys shall reside exclusively in backend environment variables and shall never be transmitted to the frontend.
- **NFR-SEC-003**: Upload endpoints shall validate file MIME types and enforce a maximum file size of 10MB per document.
- **NFR-SEC-004**: CORS shall be configured to allow only the deployed frontend origin(s) and `localhost` during development.
- **NFR-SEC-005**: All database queries shall use parameterized statements or ORM abstraction to prevent SQL injection.
- **NFR-SEC-006**: Chat and upload endpoints shall implement rate limiting (e.g., 100 requests per minute per IP) to prevent abuse.

### 5.5 Maintainability
- **NFR-MAI-001**: Backend unit and integration test coverage for services and controllers shall be at least 70%.
- **NFR-MAI-002**: The system shall include end-to-end tests for the critical user journeys: chat with RAG, document upload, and evaluation trigger.
- **NFR-MAI-003**: Prompt templates shall be stored as versioned code assets (TypeScript files) under a dedicated directory, with changes requiring pull request review.
- **NFR-MAI-004**: The backend shall expose auto-generated OpenAPI (Swagger) documentation derived from NestJS DTOs and controllers.
- **NFR-MAI-005**: The repository shall maintain clear separation between backend (`apps/api` or `/backend`) and frontend (`apps/web` or `/frontend`), with a shared types package if using a monorepo.

### 5.6 Deployability
- **NFR-DEP-001**: The frontend shall be deployable to Vercel with zero additional infrastructure configuration beyond environment variables.
- **NFR-DEP-002**: The backend shall be deployable to Render or Railway using a Node.js build command or a provided Dockerfile.
- **NFR-DEP-003**: All required environment variables shall be documented in `.env.example`, and the application shall fail fast during startup with a descriptive error if any required variable is missing.
- **NFR-DEP-004**: Database schema migrations shall be executable as an explicit pre-deployment step or automatically on application startup, with idempotency guarantees.

## 6. Acceptance Criteria by Area

### 6.1 RAG Pipeline
- **Given** a user uploads a valid PDF, TXT, or Markdown file via the UI  
  **When** the upload completes  
  **Then** a `Document` record is created with status `pending`, and an ingestion job is enqueued within 1 second.
- **Given** a document ingestion job is active  
  **When** parsing completes successfully  
  **Then** the document status transitions to `chunking`, then `embedding`, then `completed` without manual intervention.
- **Given** a document is fully ingested  
  **When** a chat query is submitted  
  **Then** the retrieval step returns at least one chunk from that document if semantically relevant.
- **Given** a document fails parsing (corrupt file)  
  **When** the worker exhausts retries  
  **Then** the document status is set to `failed`, a dead-letter record is created, and the user sees an error in the document list.

### 6.2 LLM Orchestration
- **Given** the system is configured to use Ollama  
  **When** a chat query is submitted  
  **Then** the request is routed to the local Ollama endpoint and a valid response is returned.
- **Given** the system is configured to use OpenAI  
  **When** a chat query is submitted  
  **Then** the request is routed to the OpenAI API, token usage is recorded, and latency is logged.
- **Given** a prompt template defines a JSON schema for the output  
  **When** the LLM returns a response  
  **Then** the response is validated against the schema; validation failures trigger one retry or return a structured error.
- **Given** a user submits a query exceeding the configured max length  
  **When** input guardrails run  
  **Then** the system rejects the request with HTTP 400 before calling any LLM.

### 6.3 Agent Layer
- **Given** a user asks a factual question about uploaded documents  
  **When** the Decision Agent processes the query  
  **Then** the agent selects the RAG path, triggers retrieval, and passes chunks to the synthesis workflow.
- **Given** a user asks a generic greeting ("hello")  
  **When** the Decision Agent processes the query  
  **Then** the agent selects the direct answer path and skips retrieval.
- **Given** the Synthesis Workflow is invoked  
  **When** the workflow completes  
  **Then** the final response includes an answer and a list of cited chunk IDs.
- [ ] Tool Registry contains at least two tools: Retrieval Tool and Calculator Tool.
- [ ] LangGraph state transitions are logged with correlation IDs for every step.

### 6.4 Memory & State Management
- **Given** a user has an active session with 3 prior turns  
  **When** a new message is sent  
  **Then** the LLM prompt includes the prior 3 user/assistant exchanges as context.
- **Given** a user sends an identical query within the cache TTL window  
  **When** the request is received  
  **Then** the response is served from Redis cache without invoking the LLM, and the UI displays a cache indicator.
- **Given** a session is idle for 24 hours  
  **When** the TTL expires  
  **Then** the session memory is evicted from Redis automatically.

### 6.5 Observability
- **Given** any chat request is processed  
  **When** the request completes or fails  
  **Then** a structured log entry exists containing correlation ID, prompt, response, latency, tokens, and error (if any).
- **Given** 10 chat requests have been processed  
  **When** the observability dashboard loads  
  **Then** it displays the 10 requests in a table with sortable columns and aggregate metrics (avg latency, total tokens, error rate).
- **Given** a correlation ID from a request  
  **When** an operator queries logs by that ID  
  **Then** all related events (retrieval, LLM call, agent steps) are returned in chronological order.

### 6.6 Evaluation Pipeline
- **Given** the evaluation dataset contains 25 questions  
  **When** an evaluation run is triggered for Ollama and OpenAI  
  **Then** the system produces per-model scores for relevance, consistency, grounding, latency, and tokens.
- **Given** an evaluation run completes  
  **When** the results page loads  
  **Then** the UI presents a side-by-side comparison table with sortable metrics.
- **Given** an evaluation job fails mid-run (e.g., LLM timeout)  
  **When** retries are exhausted  
  **Then** the run status is `failed`, partial results are preserved, and the failure reason is recorded.

### 6.7 Web Interface
- **Given** a user is on the chat page  
  **When** they submit a question that triggers RAG  
  **Then** the assistant response is displayed with an expandable sources panel showing retrieved chunk previews.
- **Given** a user drags a file onto the upload area  
  **When** the file is valid  
  **Then** the upload begins, a progress indicator appears, and the document list updates to show ingestion status.
- **Given** a user navigates to Settings  
  **When** they change the LLM provider and save  
  **Then** the new provider is active for the next chat request without requiring a server restart.

### 6.8 Async Job Processing
- **Given** 5 documents are uploaded simultaneously  
  **When** they are enqueued  
  **Then** ingestion jobs are processed respecting concurrency limits, and no worker exceeds its max concurrent job count.
- **Given** an embedding job fails due to a transient network error  
  **When** the failure is detected  
  **Then** the job is retried up to 3 times with exponential backoff before being marked failed.
- [ ] All queue names (`ingestion`, `embedding`, `evaluation`) are declared in a single configuration file.
- [ ] Dead-lettered jobs are queryable via API at `/api/v1/queues/dead-letter`.

## 7. Data Model

### 7.1 Entity Definitions

**Document**
- `id` (UUID, PK)
- `filename` (string) — stored file name
- `originalName` (string) — user-provided name
- `mimeType` (enum: `application/pdf`, `text/plain`, `text/markdown`)
- `sizeBytes` (integer)
- `status` (enum: `pending`, `parsing`, `chunking`, `embedding`, `completed`, `failed`)
- `metadata` (JSONB) — parser info, page count, etc.
- `createdAt` (timestamp)
- `updatedAt` (timestamp)

**Chunk**
- `id` (UUID, PK)
- `documentId` (UUID, FK → Document.id, onDelete: CASCADE)
- `content` (text) — raw chunk text
- `index` (integer) — positional order within document
- `metadata` (JSONB) — start/end offsets, section headers
- `createdAt` (timestamp)

**Embedding**
- `id` (UUID, PK)
- `chunkId` (UUID, FK → Chunk.id, onDelete: CASCADE, unique)
- `vector` (vector(768 or 1536)) — pgvector type; dimension depends on active embedding model
- `modelName` (string) — e.g., `nomic-embed-text`, `text-embedding-3-small`
- `createdAt` (timestamp)

**Session**
- `id` (UUID, PK)
- `metadata` (JSONB) — user agent, origin, optional labels
- `createdAt` (timestamp)
- `updatedAt` (timestamp)

**Message**
- `id` (UUID, PK)
- `sessionId` (UUID, FK → Session.id, onDelete: CASCADE)
- `role` (enum: `user`, `assistant`, `system`)
- `content` (text)
- `sources` (JSONB) — array of `{ chunkId, documentId, preview, score }`
- `latencyMs` (integer) — total request latency
- `tokens` (JSONB) — `{ input: number, output: number }`
- `correlationId` (string, indexed)
- `modelProvider` (string)
- `modelName` (string)
- `createdAt` (timestamp)

**EvaluationRun**
- `id` (UUID, PK)
- `datasetVersion` (string) — git hash or semver of dataset file
- `status` (enum: `pending`, `running`, `completed`, `failed`)
- `models` (JSONB) — array of `{ provider, modelName }` tested
- `aggregatedMetrics` (JSONB) — `{ avgRelevance, avgConsistency, avgGrounding, avgLatencyMs, totalTokens }`
- `startedAt` (timestamp)
- `completedAt` (timestamp, nullable)

**EvaluationResult**
- `id` (UUID, PK)
- `runId` (UUID, FK → EvaluationRun.id, onDelete: CASCADE)
- `questionId` (string) — reference to dataset question
- `questionText` (text)
- `expectedAnswer` (text, nullable)
- `generatedAnswer` (text)
- `retrievedChunkIds` (JSONB)
- `relevanceScore` (float, 0–1)
- `consistencyScore` (float, 0–1)
- `groundingScore` (float, 0–1)
- `latencyMs` (integer)
- `tokens` (JSONB)
- `createdAt` (timestamp)

**LogEntry**
- `id` (UUID, PK)
- `correlationId` (string, indexed)
- `level` (enum: `info`, `warn`, `error`, `debug`)
- `service` (string) — module or service name
- `eventType` (string) — e.g., `llm.request`, `retrieval.complete`, `agent.step`
- `payload` (JSONB) — structured event data
- `timestamp` (timestamp, indexed)

**JobRecord** (mirror of BullMQ state for queryability)
- `id` (UUID, PK)
- `queueName` (string)
- `bullJobId` (string)
- `status` (enum: `waiting`, `active`, `completed`, `failed`)
- `payload` (JSONB) — job input summary
- `attempts` (integer)
- `failedReason` (text, nullable)
- `createdAt` (timestamp)
- `updatedAt` (timestamp)

### 7.2 Relationships
- **Document 1:N Chunk** — A document contains many chunks.
- **Chunk 1:1 Embedding** — Each chunk has exactly one embedding vector.
- **Session 1:N Message** — A session contains an ordered list of messages.
- **EvaluationRun 1:N EvaluationResult** — A run produces one result per dataset question per model.

## 9. Interaction Patterns & Accessibility

### 9.1 Non-Negotiable Rules

These rules apply to every frontend task. Breaking them requires explicit justification in the PR:

1. **Canvas is black (`#000`) or near-black (`surface-1: #111`).** Never use white as a main surface background.
2. **Only one primary brand color: lime `#C5F547`.** Used for CTAs, active/selected states, featured metric highlights, success indicators. Never use lime on large areas (>30% of viewport).
3. **Purple is reserved for "AI/IA".** Orbs, insight gradients, assistant states. Never use purple for generic UI.
4. **Hierarchy by weight, not size.** Headings use weight 400 (Regular) with 600 (Semibold) highlights. Never use 700+ in display.
5. **Instrument Serif (italic) is rare and expressive.** Only for one keyword in cover/hero headings. Never in functional UI.
6. **Monospace only for technical data:** IDs, timestamps, paths, numeric metrics, code, correlation IDs. Never for UI labels, body, buttons, nav.
7. **Generous border radius:** Buttons and inputs are pills (`9999px`). Cards are `20px` (`r-lg`) or `28px` (`r-xl`). Never `4px` or `8px` on cards.
8. **Icons: lucide-style line icons, stroke 1.75, 16px default.** Never mix styles.
9. **Never hardcode colors.** Use CSS tokens. If a variation is needed, use `oklch()` or opacity over an existing token.
10. **No emoji in functional UI.** Only in specific demo/marketing copy.

### 9.2 Design Tokens in Interactions

All interactive states must reference the token system:

| State | Token Application |
|-------|-------------------|
| **Hover** | Brightness filter `1.05–1.1`, or background shift to next surface level (`surface.1` → `surface.2`) |
| **Active / Press** | Scale `0.98`, duration `instant` (80ms) |
| **Focus Visible** | `outline: 2px solid brand.lime.400; outline-offset: 2px` |
| **Disabled** | `text.disabled` + `opacity: 0.5`, no pointer events |
| **Selected** | `brand.lime.400` bg for pills; `brand.purple.500` left border for cards |

### 9.2 Component Interaction Patterns

**Pill / Badge**
- Hover: background lightens one step; cursor pointer
- Active (selected): `brand.lime.400` bg, black text, no border
- Filter pills: toggle on click; multi-select allowed where specified

**Button**
- Primary: hover `background: var(--lime-500)` (slightly darker), active `scale(0.98)`, focus visible lime outline
- Secondary/Ghost: hover `border: var(--border-strong)`
- Icon Button: hover `background: var(--surface-3)`; active `scale(0.95)`
- Ghost: hover `surface.1` bg; used for low-emphasis actions

**Metric Card**
- **Cards do not change on hover** (avoids noise).
- On mount: metric counter animates `0 → value` in ~800ms with `decelerate` easing
- Delta chip: positive = `status.success` (`#C5F547`), negative = `status.danger` (`#F47266`), neutral = `text.muted`
- Sparkline (if present): draws left-to-right over 640ms `deliberate`

**Chat Input**
- Focus: container border shifts to `border.strong`
- Send Button: disabled until text present; enabled state transitions with `fast` (150ms) opacity
- File attachment: dropping a file triggers a brief `glowLime` shadow on the input

**Citation Card**
- Hover: `border.default` → `border.strong`, thumbnail brightness increase
- Click: opens Sources Panel and scrolls to chunk; chunk receives 2px `brand.lime.400` ring for 2s
- Keyboard: focusable `<button>` with `aria-describedby` pointing to source chunk ID

**Orb AI / MiniOrb**
- Default: continuous pulse `scale(1) → 1.04 → 1` over 2.4s infinite
- Thinking state: pulse speed increases to 1.2s
- Reduced motion: pulse disabled; static sphere displayed
- **Never** use orb for non-AI indicators.

**Accordion / Drawer**
- Expand/collapse: height animation with `base` (220ms) `standard` easing
- Focus trap required inside drawers; `Escape` closes
- On close: focus returns to trigger element

**List Rows**
- Hover: `background: var(--surface-2)`
- Selected: `border-left: 2px solid var(--lime-400)` + `background: var(--surface-2)`

**Inputs**
- Focused: `border: 1.5px solid var(--border-strong)`. No white ring.
- Selection cards (e.g., provider in Settings): `border: 1.5px solid var(--lime-400)` when active.

### 9.3 Layout Patterns

**Dashboard Grid**
- **Columns**: 12-column grid, gap `24px`
- **Breakpoint behavior**:
  - `≥1440px`: full 12-column layout
  - `<1024px`: 2-column layout
  - `<768px`: 1-column stack
- **Cards**: always span integer multiples of columns; never odd partial spans that break alignment

**Hero**
- Large display heading with mixed weights (Regular + Semibold)
- Sub-description (`text.secondary`, max 60ch)
- Pill row (1–3 pills: section label, status, optional CTA)
- Product mockup or Orb AI visual right-aligned or centered below

**Section Intro**
- Sequence: Pill label (`brand.lime.400` or `brand.purple.400`) → large heading with Semibold emphasis → optional **Figma ornament** (purple bounding box with handles) → body text (`text.secondary`)

**Pipeline Timeline**
- Zigzag card layout with colored tiles (`cat.*` tokens) and 1px dashed connecting lines
- Each tile: icon in circular container + stage name + status pill
- Completed stages: solid tile; in-process: hatched pattern overlay; pending: `text.disabled`

**Problem / Solution Layout**
- Solution is always visually dominant: large lime card (`brand.lime.400` bg, black text)
- Problem is plain text (`text.secondary`) on `bg.canvas`
- Ratio: ~60/40 or 70/30 in favor of the solution card

### 9.4 Assets & Micro-interactions

**Star Particles**
- Trigger: always visible on Orb AI and Featured AI Card
- Animation: each dot oscillates opacity independently (randomized 2–3s cycle) to avoid mechanical uniformity
- Reduced motion: opacity oscillation disabled; static dots at 60% opacity

**Hatched Patterns**
- Trigger: applied to skeleton loaders, in-process job cards, and placeholder chart states
- Animation: optional slow diagonal drift (`translateX`) on loading states to suggest activity
- Reduced motion: drift disabled; static pattern

**Metric Counters**
- Trigger: on mount or when entering viewport (`IntersectionObserver`)
- Animation: count from 0 to target value over ~800ms using `decelerate` easing
- Reduced motion: value appears instantly

**Toast Notifications**
- **Position**: top-right stack, max 3 visible
- **Enter**: slide from right + fade, `fast` (150ms) `emphasized` easing
- **Exit**: fade out, `instant` (80ms)
- **Duration**: Success = 4s, Info = 6s, Error = persistent until dismissed
- **Actions**: Error toasts include a ghost "Retry" Button; success toasts for long jobs include a "View" link
- **Accessibility**: `aria-live="polite"` region; toasts are focusable and dismissible via `Escape`

**Drag-and-Drop Upload**
- **Default**: `2px dashed border.subtle`, `surface.2` bg
- **Dragover**: border shifts to `brand.purple.500`, `glowPurple` shadow, label changes to "Release to upload"
- **Drop**: brief `glowLime` flash, then transitions to Pipeline Timeline
- **Multi-file**: 5+ files collapse into summary "5 files selected" with expand chevron
- **Validation**: invalid files flagged with `status.danger` row at top of file list; submission blocked until removed

**Real-Time Status Updates**
- **Mechanism**: Server-Sent Events (SSE) for ingestion jobs and evaluation runs; fallback to 3s short polling
- **Progress**: smooth width transition (`transition-all duration-500`, `base` easing)
- **Step completion**: checkmark icon scales from 0→1 with `fast` (150ms) `emphasized` easing
- **Completion**: toast triggers with lime accent + delta chip if applicable

**Side-by-Side Comparison (Chat)**
- **Desktop**: 50/50 split with draggable vertical divider (min 280px per pane). Divider handle: 4px `border.strong`, hover `brand.lime.400`
- **Tablet**: stacked layout with toggle pill ("Model A" / "Model B")
- **Mobile**: comparison mode disabled; button shows tooltip "Requires larger screen"
- **Synchronization**: question thread scroll is shared; citations in each pane only highlight that pane's Sources Panel

### 9.6 Accessibility Requirements (Dark Mode & High Contrast)

| Requirement | Implementation |
|-------------|----------------|
| **Color Contrast** | `text.primary` (#FFF) on `bg.canvas` (#000) = 21:1. `text.secondary` (#C7C7C7) on `surface-1` (#111) = 11.5:1. `text-muted` (#8A8A8A) on `surface-1` = 5.3:1 (AA only). `text-on-lime` (#0A0A0A) on `lime-400` (#C5F547) = 14.8:1. **Never** use `text-muted` on `surface-3` — insufficient contrast. |
| **Focus Visible** | `outline: 2px solid var(--lime-400); outline-offset: 2px`. Never suppress focus rings. |
| **Keyboard Navigation** | All interactive elements reachable via `Tab`. Modal traps focus; `Escape` closes modals/drawers. Focus returns to trigger on close. |
| **Hit Targets** | Buttons and nav: minimum 40px height. Mobile tab bar: 44×44. Inline table links: extra padding to reach 32px minimum. |
| **Screen Reader Support** | Landmark regions (`<main>`, `<nav>`, `<aside>`). Live regions for streaming text updates and status changes. Icons have `aria-hidden="true"` with visible text labels or `aria-label`. `<Pill>` with informative status: `role="status"`. `<Orb>` decorative: `aria-hidden="true"`. Dense tables: `<caption>` required. |
| **Status Without Color** | Never use color alone for status. Always pair with lucide icon + text label (e.g., warning triangle + "Failed"). |
| **Reduced Motion** | Respect `prefers-reduced-motion`: disable Orb AI pulse, metric counters, toast slide-in, skeleton pulse, star particle oscillation, and progress bar shimmer. Use instant opacity transitions instead. |
| **Text Scaling** | Layout supports 200% browser zoom without horizontal scroll on primary containers. Sidebar collapses automatically at zoom levels that constrain width. |
| **Form Labels** | All inputs have visible labels (not just placeholders). Chunking sliders announce values via `aria-valuenow`. |
| **Loading & Streaming** | Loading states announced via `aria-busy` and `aria-live`. Streaming text updates the live region per sentence to prevent excessive chatter. |
| **Dark Mode** | System is dark-only (`bg.canvas` = `#000000`). No light mode toggle. All tokens are optimized for pure-black canvas. |

### 9.7 Icon System Rules

- **Library**: `lucide-react` exclusively. No emojis or custom SVG icons except the logo monogram.
- **Container Rule**: Every icon must live inside a circular container (32px or 40px diameter) with a contrasting background (`surface.2` or categorical color at 10% opacity). This ensures visibility on `bg.canvas`.
- **Sizing**: 20px icon inside 40px container; 16px icon inside 32px container.
- **Color**: Icon stroke uses `text.primary`, `text.secondary`, or the categorical color of the parent card.
- **Accessibility**: Icons are decorative unless standalone; always use `aria-hidden="true"` + visible text label, or `aria-label` if text is visually hidden.
- **Status Icons**: `CheckCircle` (success, lime), `AlertTriangle` (warning, yellow), `XCircle` (danger, rose), `Info` (info, purple). Always paired with text.
